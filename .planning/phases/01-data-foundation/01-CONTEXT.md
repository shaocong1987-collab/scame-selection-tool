# Phase 1 Context: 数据基础构建

## Phase Goal
建立100%精准的SCAME产品数据基础，彻底解决串号/幻觉问题，为工业安全级选型提供可靠数据支撑。

## Success Criteria (必须满足)
1. **数据完整性**：12000行SCAME产品数据完全结构化入库，无缺失字段
2. **规则准确性**：编码规则引擎100%准确解析任意SCAME订货号，零误差
3. **验证完备性**：产品参数验证体系完备，禁止任何参数错位或矛盾
4. **安全就绪**：数据加密存储和多租户隔离机制就绪，符合工业数据安全要求

## Scope Boundaries

### In Scope (本次实现)
- SCAME全系列产品数据结构化（插头、插座、连接器、附件等）
- 编码规则引擎核心功能实现
- 基础数据验证逻辑
- 数据库设计和API接口
- 数据加密和访问控制基础

### Out of Scope (后续阶段)
- 选型业务逻辑实现 (Phase 2)
- 用户界面优化 (Phase 2+)
- 高级搜索功能 (Phase 2)
- 企业集成功能 (Phase 5)

## Technical Constraints

### 刚性约束 (不可突破)
1. **安全约束**：产品数据必须100%准确，禁止任何可能导致选型错误的数据问题
2. **性能约束**：编码规则解析响应时间 < 100ms
3. **兼容约束**：必须与现有前端原型无缝集成
4. **成本约束**：采用轻量级技术栈，控制云服务成本

### 技术选型偏好
1. **数据库**：PostgreSQL (关系型数据) + Redis (缓存)
2. **规则引擎**：JSONLogic 或自定义TypeScript规则引擎
3. **API框架**：Next.js API Routes (全栈) 或 Express.js
4. **部署平台**：Vercel (前端) + Railway (后端)

## Key Decisions Needed

### 待决策项
1. **规则引擎技术选型**：JSONLogic vs 自定义TypeScript引擎
   - JSONLogic: 声明式，易维护，但性能可能受限
   - 自定义引擎: 高性能，紧密集成，但开发成本高

2. **数据库架构**：单库多租户 vs 多库隔离
   - 单库多租户: 开发简单，成本低
   - 多库隔离: 安全性高，扩展性好

3. **数据加密策略**：字段级加密 vs 数据库级加密
   - 字段级加密: 细粒度控制，性能开销大
   - 数据库级加密: 简单，依赖数据库特性

### 已决策项
1. **PostgreSQL作为主数据库**：关系型数据适合产品目录
2. **TypeScript作为主要开发语言**：与前端技术栈一致
3. **RESTful API设计**：便于前后端分离和第三方集成

## Dependencies

### 外部依赖
1. **SCAME产品数据源**：wiki知识图谱，需要数据访问权限
2. **SCAME官方文档**：编码规则验证参考
3. **云服务平台**：Vercel, Railway 账号准备

### 内部依赖
1. **现有前端原型**：需要API接口定义一致
2. **现有编码规则实现**：`src/lib/scame/coding.ts` 需要扩展和完善

## Risk Assessment

### 高风险
1. **数据准确性风险**：产品数据可能不完整或有误
   - **缓解措施**：建立多层验证机制，人工审核关键数据

2. **规则复杂度风险**：SCAME编码规则可能过于复杂
   - **缓解措施**：分阶段实现，先核心后边缘规则

### 中风险
1. **性能风险**：12000行数据处理可能影响性能
   - **缓解措施**：数据库优化，索引设计，分批处理

2. **集成风险**：前后端API接口可能不匹配
   - **缓解措施**：API契约先行，接口测试覆盖

## Phase Exit Criteria

本阶段完成时，必须满足：
- [ ] 所有12000行产品数据可查询、可验证
- [ ] 编码规则引擎通过1000+测试用例
- [ ] 数据API接口文档完整
- [ ] 安全机制通过基础渗透测试
- [ ] 性能基准测试达标
