# AI优先项目开发指南

## 概述

本指南定义了基于新安装的23个AI技能的**标准化开发流程**，遵循「AI优先工作流」：需求→AI梳理→框架生成→约束验证→交付验收。本流程确保：
1. **用户知情权**：用户清楚每个阶段在做什么
2. **无意外代码**：未经用户批准绝不开始编码
3. **质量保证**：通过严格审查机制确保交付质量
4. **知识沉淀**：每个项目产出完整文档

## 🎯 核心原则

### 角色分工
- **用户（韶聪）**：产品负责人，负责需求定义、功能确认、用户体验验收
- **AI助手（Claude）**：技术执行者，负责技术方案、代码实现、质量保证

### 决策权归属
- **功能需求**：用户拥有最终决定权
- **技术方案**：AI提供专业建议，用户确认方向
- **代码实现**：AI负责执行，用户监督进展

## 📋 五阶段开发流程

### 阶段1：需求澄清（用户决策阶段）

**目标**：明确"要做什么"和"为什么要做"
**使用技能**：`brainstorming`（Superpowers核心技能）
**触发时机**：任何新功能、重大修改、项目启动时

#### 用户职责
1. 描述需求和期望
2. 回答澄清问题（通常通过选择题）
3. 从2-3个方案中选择方向
4. **批准最终设计方案**

#### AI职责
1. 使用brainstorming技能进行深度分析（苏格拉底式追问）
2. 提出2-3个合理方案，包含优缺点对比
3. 产出设计方案文档
4. 达到95%置信度后再输出方案

#### 决策点
🔴 **必须获得用户书面批准才能进入下一阶段**
- 设计方案必须清晰、无歧义
- 所有TBD/TODO必须被明确回答
- 技术可行性得到验证

#### 产出文档
`docs/design/YYYY-MM-DD-项目名-design.md`
- 问题定义与背景
- 目标用户与使用场景
- 功能需求清单
- 技术方案对比
- 风险评估与缓解措施

---

### 阶段2：方案设计（技术规划阶段）

**目标**：明确"怎么做"的技术实现方案
**使用技能**：`writing-plans`（Superpowers技能）
**触发时机**：需求澄清阶段完成后

#### 用户职责
1. 确认技术方案符合业务需求
2. 提出用户体验改进建议
3. 确认验收标准

#### AI职责
1. 将设计方案拆解为可执行任务（每个任务2-5分钟可完成）
2. 创建详细实施计划
3. 预估工作量和风险
4. 设计测试验证方案

#### 关键原则
- **初级工程师可执行**：计划需足够详细，让没有上下文的工程师也能照着做
- **任务原子化**：每个任务完成后立即验证，避免整体完成后才发现问题
- **依赖清晰**：明确任务间的依赖关系

#### 产出文档
`docs/plan/YYYY-MM-DD-项目名-plan.md`
- 任务分解清单
- 技术实现细节
- 依赖关系图
- 验收标准矩阵
- 风险应对预案

---

### 阶段3：开发执行（AI工作阶段）

**目标**：编写和测试代码
**使用技能**：
- `using-git-worktrees`（创建隔离开发环境）
- `subagent-driven-development`（并行开发）
- `dispatching-parallel-agents`（任务调度）
- `frontend-design`（前端界面设计）
- `web-access`（网页抓取/数据获取）
- `executing-plans`（计划执行）

**触发时机**：方案设计阶段完成后

#### 用户职责
1. 阶段性查看进展（非必须但欢迎）
2. 提供功能反馈（如发现问题）
3. 保持沟通渠道畅通

#### AI职责
1. 创建隔离git工作区（using-git-worktrees）
2. 按计划实现功能
3. 定期汇报进度（每完成一个重要里程碑）
4. 遇到问题及时沟通
5. **绝不擅自修改需求或范围**

#### 特殊情况处理
- **前端界面开发**：必须使用`frontend-design`技能，避免AI审美，确定统一美学方向
- **数据抓取需求**：使用`web-access`技能，支持登录状态保持
- **AI摆烂情况**：使用`pua`技能强制执行检查清单

#### 产出物
- 可运行代码
- 单元测试和集成测试
- 临时部署验证环境（如需要）

---

### 阶段4：审查验证（质量保证阶段）

**目标**：确保代码质量和功能正确性
**使用技能**：
- `requesting-code-review`（代码质量审查）
- `receiving-code-review`（全局架构审查）
- `verification-before-completion`（最终验证）
- `systematic-debugging`（系统化调试）
- `test-driven-development`（测试驱动开发支持）

**触发时机**：开发执行阶段完成后

#### 用户职责
1. 验收功能是否符合需求
2. 测试用户体验
3. 提出改进建议
4. **最终批准交付**

#### AI职责
1. 执行三级审查机制：
   - 第一层：代码质量审查（规范性、可维护性）
   - 第二层：功能正确性审查（按需求实现）
   - 第三层：全局架构审查（模块集成、一致性）
2. 修复问题和bug
3. 优化性能和体验
4. 准备交付物

#### 审查标准
1. **语法/格式**：通过所有lint检查
2. **逻辑/边界**：通过所有测试用例
3. **安全/性能**：通过安全扫描和性能测试
4. **用户体验**：符合设计规范，交互流畅

#### 产出文档
`docs/review/YYYY-MM-DD-项目名-review.md`
- 审查发现的问题清单
- 修复方案和验证结果
- 性能测试报告
- 安全扫描报告

---

### 阶段5：交付部署（项目完成阶段）

**目标**：完成项目交付
**使用技能**：`finishing-a-development-branch`
**触发时机**：审查验证阶段完成后

#### 用户职责
1. 最终验收和确认
2. 部署到目标环境（如需要）
3. 更新项目状态文档

#### AI职责
1. 合并代码到主分支
2. 清理临时文件和隔离环境
3. 生成最终项目文档
4. 部署到服务器（如需要）
5. 创建发布说明

#### 交付物标准
✅ **代码交付物**
- 可运行代码 + 完整测试套件（覆盖率>90%）
- 自动化部署脚本 + 监控配置
- 性能基准报告 + 安全扫描报告

✅ **文档交付物**
- 架构决策记录（ADR）
- 用户使用手册（傻瓜式）
- 运维故障排除手册
- API接口文档（可交互式）

✅ **知识交付物**
- 可复用代码模板
- 最佳实践指南
- 常见问题解决方案库

#### 产出文档
`docs/delivery/YYYY-MM-DD-项目名-delivery.md`
- 发布说明
- 部署指南
- 运维手册
- 回滚方案

## 🛡️ 关键保障机制

### 1. 无意外代码保障
**规则**：阶段1设计方案未批准前，AI绝不会开始写代码
**实现**：
- 所有编码工作从阶段3开始
- 阶段1和阶段2只产出文档和计划
- 用户明确看到"开发执行阶段开始"的提示

### 2. 透明沟通机制
**规则**：每个阶段开始和结束都会明确告知用户
**实现**：
- 阶段转换时使用清晰标题标注
- 定期进度汇报（至少每完成一个里程碑）
- 遇到问题立即沟通，不隐藏困难

### 3. 用户决策优先
**规则**：用户拥有最终决定权，可随时暂停或调整方向
**实现**：
- 每个重要决策点都需要用户确认
- 任何时候用户说"暂停"或"重新讨论"，立即停止当前工作
- 提供"回退到上一阶段"的机制

### 4. 质量三重保障
**规则**：通过多层审查确保交付质量
**实现**：
- 开发过程中的自检（语法、逻辑、安全）
- 专门的审查验证阶段
- 用户最终验收测试

## 🛠️ 技能使用指南

### 优先使用的新安装技能（2026年4月15日安装）

#### Superpowers工作流核心技能
1. **brainstorming** - 深度需求分析（阶段1必须使用）
2. **writing-plans** - 任务拆解（阶段2必须使用）
3. **using-git-worktrees** - 隔离开发环境（阶段3推荐使用）
4. **subagent-driven-development** - 并行开发（阶段3推荐使用）
5. **requesting-code-review/receiving-code-review** - 多层审查（阶段4必须使用）
6. **verification-before-completion** - 最终验证（阶段4必须使用）
7. **finishing-a-development-branch** - 交付合并（阶段5必须使用）

#### 第二篇文章推荐技能
8. **frontend-design** - 前端界面设计（所有前端开发必须使用）
9. **web-access** - 网页抓取/浏览器操作（数据获取场景必须使用）
10. **pua** - AI摆烂问题解决（遇到困难时使用）
11. **mem-search/knowledge-agent** - 记忆与知识管理（参考历史经验时使用）
12. **docx/xlsx/pdf/pptx** - Office文档处理（文档生成场景必须使用）

### 技能触发规则
1. **默认优先新技能**：当多个skills可能适用时，优先选择新安装的skills
2. **强制使用场景**：
   - 前端界面设计 → 使用 `frontend-design`
   - Office文档处理 → 使用 `docx`, `xlsx`, `pdf`, `pptx`
   - 联网搜索/浏览器操作 → 使用 `web-access`
   - AI摆烂/深度调试 → 使用 `pua`

### 技能组合最佳实践
- **前端开发组合**：`frontend-design` + `using-superpowers`完整工作流
- **文档处理组合**：办公四件套 + `skill-creator`（个性化模板）
- **复杂问题解决**：`pua` + `systematic-debugging` + `mem-search`
- **完整项目开发**：`using-superpowers`完整工作流 + `frontend-design` + 办公四件套

## 🔄 异常处理流程

### 需求变更处理
1. **微小调整**：在当前阶段内处理，更新相关文档
2. **中等变更**：回退到阶段1重新进行需求澄清
3. **重大变更**：作为新项目重新启动整个流程

### 技术问题处理
1. **预期内问题**：按计划中的风险应对预案处理
2. **未预期问题**：
   - 立即向用户汇报
   - 评估影响范围和解决方案
   - 获得用户同意后继续或调整方案

### 时间/资源超出预期
1. **轻微超出**：调整后续任务优先级
2. **严重超出**：
   - 重新评估项目范围和计划
   - 与用户讨论调整方案（简化功能、分阶段交付等）
   - 更新相关文档和计划

## 📚 文档管理规范

### 文档结构
```
docs/
├── design/                 # 阶段1产出：设计方案
│   └── YYYY-MM-DD-项目名-design.md
├── plan/                  # 阶段2产出：实施计划  
│   └── YYYY-MM-DD-项目名-plan.md
├── review/               # 阶段4产出：审查报告
│   └── YYYY-MM-DD-项目名-review.md
├── delivery/             # 阶段5产出：交付文档
│   └── YYYY-MM-DD-项目名-delivery.md
└── knowledge/            # 知识沉淀
    └── 项目名-best-practices.md
```

### 文档更新规则
1. **实时更新**：每个阶段完成后立即更新对应文档
2. **版本控制**：重要文档变更需创建新版本
3. **关联更新**：上游文档变更时，下游文档需相应更新

### 文档质量要求
- **清晰性**：技术小白也能理解核心内容
- **完整性**：无TBD/TODO占位符
- **实用性**：直接指导下一步行动
- **可维护性**：结构清晰，易于更新
